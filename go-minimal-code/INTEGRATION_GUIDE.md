# Go Minimal Code 集成指南

## 概述

Go Minimal Code（原 Ponytail）行为矫正规则集已成功集成到项目技能体系中，作为**过度工程防御层**，与其他专项审查技能形成互补。

## 集成架构

```
项目技能体系
├── 错误处理审查 (error-handling-reviewer)
│   └── ✨ 集成 Go Minimal Code：过度工程审查维度
│       ├── YAGNI 错误处理
│       ├── 错误包装深度限制
│       ├── 删除冗余错误转换
│       └── 标准化错误处理模式
│
├── 依赖管理审查 (dependency-reviewer)
│   └── ✨ 集成 Go Minimal Code：标准库优先强化
│       ├── 标准库替代优先级检查
│       ├── 可疑依赖特征识别
│       ├── 依赖必要性评估模板
│       └── Go Minimal Code 式依赖决策
│
├── 工具函数封装 (go-utility-functions)
│   └── ✨ 集成 Go Minimal Code：封装 YAGNI 判断
│       ├── 封装决策树
│       ├── 封装门槛（重复次数要求）
│       └── 封装收益 vs 维护成本评估
│
└── 新功能开发 (feature-development-workflow)
    └── 📋 待集成：开发前 YAGNI 判断环节
```

## 各技能集成详情

### 1. 错误处理审查员 (error-handling-reviewer)

**新增能力**：在审查错误处理的同时，识别过度工程

**审查维度扩展**：
- **原有**：被忽略的 error、panic 滥用、错误包装、参数校验
- **新增**：未请求的错误类型层次、错误包装过深、冗余错误转换

**典型发现**：
```go
// 🔴 Go Minimal Code 发现：过度设计的错误类型
type AppError struct {
    Code    int
    Message string
    Cause   error
}

// ✅ Go Minimal Code 建议：先用标准 error
return fmt.Errorf("business logic failed: %w", err)
```

### 2. 依赖管理审查员 (dependency-reviewer)

**新增能力**：引入依赖前的 YAGNI 检查

**审查维度扩展**：
- **原有**：必要性审查、依赖状态、版本锁定、传递依赖
- **新增**：标准库替代优先级、可疑依赖特征、依赖必要性评估模板

**决策流程**：
```
引入依赖前必须经过：
1. 标准库能做到吗？
2. 项目已有依赖能做到吗？
3. 一行代码能替代吗？
4. 真的需要吗？（不是"可能用到"）
```

**红旗依赖特征**：
- "工具库"型依赖（什么都做，什么都不精）
- 版本超过 2 年未更新
- 传递依赖 > 10 个
- 仅用于一个函数

### 3. 工具函数封装 (go-utility-functions)

**新增能力**：封装决策树和门槛检查

**审查维度扩展**：
- **原有**：标准库优先、泛型优先、接口最小化
- **新增**：封装 YAGNI 判断、重复次数门槛、封装收益评估

**封装门槛**：
| 重复次数 | 建议 | 说明 |
|----------|------|------|
| 1 次 | ❌ 不封装 | 第一次做决定是好抽象 |
| 2 次 | ⚠️ 谨慎 | 可能是巧合，等等看第 3 次 |
| 3 次+ | ✅ 封装 | 模式已确立，值得抽象 |

**关键原则**：
> 封装的收益必须大于维护成本。如果封装后工具函数比原始代码还长，或者需要复杂的文档才能使用，说明封装过早或过度。

## 使用场景

### 场景 1：新功能开发

**流程**：
1. **需求分析**：先用 Go Minimal Code 阶梯判断是否真的需要开发
2. **依赖选择**：优先标准库，再考虑已有依赖
3. **错误处理**：统一模式，不过度设计
4. **工具封装**：重复 3 次以上再封装

**示例**：
```
用户："我需要为 API 添加缓存"

❌ 传统方式：
   - 引入缓存库
   - 设计缓存策略
   - 编写缓存管理类

✅ Go Minimal Code 方式：
   - 检查标准库：@lru_cache 或 sync.Map
   - 一行代码解决问题
   - 等性能分析证明需要时才扩展
```

### 场景 2：代码审查

**流程**：
1. **正确性审查**：error-handling-reviewer + security-reviewer
2. **复杂度审查**：go-minimal-code（识别过度工程）
3. **依赖审查**：dependency-reviewer（检查非必要依赖）

**输出示例**：
```markdown
## 审查报告

### 正确性问题
- L123: panic 处理业务错误

### 过度工程问题 (Go Minimal Code)
- L45-78: yagni: 为单一实现定义接口
- L102: stdlib: 手写的 CSV 解析，encoding/csv 一行搞定
- L200: delete: 推测性功能开关，暂时不需要

### 依赖问题
- github.com/unused/pkg: 仅用于一个函数，建议内联
```

### 场景 3：重构优化

**流程**：
1. **识别冗余**：go-minimal-code 扫描整个代码库
2. **删除优先**：删除 > 修改 > 添加
3. **简化抽象**：接口只有一个实现 → 内联

## 与现有技能的协同效应

### 1. 错误处理 + Go Minimal Code
- **协同点**：错误处理是过度工程的重灾区
- **效果**：避免为每个错误定义自定义类型，保留错误链

### 2. 依赖审查 + Go Minimal Code
- **协同点**：依赖引入是最常见的过度工程
- **效果**：标准库优先，减少 20%+ 传递依赖

### 3. 工具封装 + Go Minimal Code
- **协同点**：过早封装是维护负担的主要来源
- **效果**：重复 3 次以上再封装，减少 30%+ 工具函数

### 4. 安全审查 + Go Minimal Code
- **协同点**：安全不可简化，但实现可以简化
- **效果**：安全机制完整，实现代码最小化

## 注意事项

### 不适用场景

以下情况**不应**启用 Go Minimal Code 原则：
1. **安全关键代码**：安全机制必须完整，不可简化
2. **合规要求严格的场景**：审计跟踪、日志保留等
3. **团队协作初期**：需要先建立模式再简化
4. **原型验证阶段**：快速实现优先，简化后续再做

### 必须保留

无论 Go Minimal Code 强度多高，以下元素**不得删除**：
- 输入验证（信任边界）
- 错误处理（防止数据丢失）
- 安全机制（认证、授权、加密）
- 用户明确要求的功能
