# Go Minimal Code 审查清单

## 概述

本检查清单用于在代码审查、重构和日常开发中系统性识别过度工程。可与任何专项审查技能配合使用。

## 检查维度

### 1. YAGNI 功能检查

| # | 检查项 | 红旗信号 | Go Minimal Code 建议 |
|---|--------|----------|---------------------|
| 1.1 | 推测性功能 | "可能以后会用到" | 删除，需要时再加 |
| 1.2 | 未使用的配置 | Config 结构体有 10 个字段，只用了 2 个 | 移除未使用字段 |
| 1.3 | 过度抽象 | 接口只有一个实现 | 内联为具体类型 |
| 1.4 | 空壳函数 | `func process() { doProcess() }` | 直接调用 `doProcess()` |

### 2. 标准库优先检查

| # | 检查项 | 红旗信号 | Go Minimal Code 建议 |
|---|--------|----------|---------------------|
| 2.1 | JSON 处理 | 手写 JSON 解析 | 用 `encoding/json` |
| 2.2 | CSV 解析 | 引入第三方 CSV 库 | 用 `encoding/csv` |
| 2.3 | HTTP 客户端 | 手写 HTTP 请求 | 用 `net/http` |
| 2.4 | 随机数 | 手写随机数生成 | 用 `crypto/rand` |
| 2.5 | 排序 | 手写排序算法 | 用 `sort` 包 |

### 3. 代码量审查

| # | 检查项 | 红旗信号 | Go Minimal Code 建议 |
|---|--------|----------|---------------------|
| 3.1 | 函数长度 | 单函数 >50 行 | 拆分为小函数 |
| 3.2 | 文件长度 | 单文件 >300 行 | 拆分模块 |
| 3.3 | 结构体字段 | 结构体 >8 个字段 | 拆分为多个结构体 |
| 3.4 | 参数数量 | 函数 >4 个参数 | 使用选项模式或结构体 |

### 4. 错误处理审查

| # | 检查项 | 红旗信号 | Go Minimal Code 建议 |
|---|--------|----------|---------------------|
| 4.1 | 错误类型层次 | 自定义 `AppError`、`BizError`、`SysError` | 先用 `fmt.Errorf` |
| 4.2 | 错误包装深度 | 嵌套 3 层以上 | 一层包装 |
| 4.3 | 错误转换 | `if err != nil { return ErrSomething }` | 保留原始错误 |
| 4.4 | 被忽略的 error | `result, _ := DoSomething()` | 必须处理 error |

### 5. 依赖审查

| # | 检查项 | 红旗信号 | Go Minimal Code 建议 |
|---|--------|----------|---------------------|
| 5.1 | 标准库替代 | 手写了 `encoding/json` 能做的事 | 改用标准库 |
| 5.2 | 已有依赖复用 | 项目已有包又引入相似包 | 复用已有依赖 |
| 5.3 | 依赖必要性 | 引入 5000 行库只用 1 个函数 | 手写 20 行替代 |
| 5.4 | 传递依赖 | `go mod graph` 显示 >20 个间接依赖 | 检查是否可精简 |

### 6. 架构审查

| # | 检查项 | 红旗信号 | Go Minimal Code 建议 |
|---|--------|----------|---------------------|
| 6.1 | 分层过度 | Request → Controller → Service → Repository | 合并为 2-3 层 |
| 6.2 | 事件总线 | 简单操作用事件驱动 | 直接调用 |
| 6.3 | 观察者模式 | 一个函数触发 5 个回调 | 合并逻辑 |
| 6.4 | 工厂模式 | `NewX()` 只做 `return &X{}` | 直接 `&X{}` |

## 审查流程

### 轻量审查（PR 审查）

```
1. 浏览改动文件
2. 标记红旗信号（使用上方表格）
3. 输出一行发现：位置 + 标签 + 建议
4. 估计净减少行数
```

### 完整审查（重构前）

```
1. 运行 `go mod graph` 检查依赖
2. 统计代码量（LOC、函数数、结构体数）
3. 逐项检查上方 6 个维度
4. 生成详细报告
5. 制定重构计划（删除优先于修改）
```

```markdown
## Go Minimal Code 审查报告

**项目**: [项目名称]
**日期**: [审查日期]
**强度**: [lite/full/ultra]

### 摘要

| 维度 | 发现问题 | 预计净减少 |
|------|----------|------------|
| YAGNI 功能 | X | -Y 行 |
| 标准库替代 | X | -Y 行 |
| 过度抽象 | X | -Y 行 |
| 错误处理 | X | -Y 行 |
| 依赖 | X | -Y 行 |
| 架构 | X | -Y 行 |
| **合计** | **X** | **-Y 行** |

### 详细发现

#### 🔴 高优先级（立即删除）

| 位置 | 问题 | Go Minimal Code 建议 | 净减少 |
|------|------|---------------------|--------|
| `foo.go:42` | 推测性功能：用户权限管理 | 删除，需要时再加 | -150 行 |
| `bar.go:10` | 手写字典排序 | 用 `sort.MapKeys()` | -30 行 |

#### 🟡 中优先级（下次重构）

| 位置 | 问题 | Go Minimal Code 建议 | 净减少 |
|------|------|---------------------|--------|
| `baz.go:55` | 错误包装 3 层深 | 一层包装 + 上下文 | -10 行 |

#### 💡 低优先级（可选优化）

| 位置 | 问题 | Go Minimal Code 建议 | 净减少 |
|------|------|---------------------|--------|
| `qux.go:20` | 函数 5 个参数 | 选项模式 | -5 行 |

### 结论

**净减少**: -XX 行（预计 -X% 代码量）

**下一步**:
1. [ ] 删除高优先级项目
2. [ ] 评估中优先级项目
3. [ ] 验证测试通过
4. [ ] 重新运行审查确认
```

## 强度级别对照

| 级别 | 行为 | 适用场景 |
|------|------|----------|
| **lite** | 提示红旗信号，用户决定是否删除 | 学习阶段、首次审查 |
| **full** | 标记所有红旗，建议删除 | 日常 PR 审查 |
| **ultra** | 激进删除，只保留核心功能 | 重构清理、技术债偿还 |

## 快速参考卡

### 删除优先清单

1. ❌ 推测性功能 → 删除
2. ❌ 未使用的配置 → 删除
3. ❌ 单一实现的接口 → 内联
4. ❌ 空壳函数 → 直接调用
5. ❌ 冗余错误类型 → 用 `fmt.Errorf`
6. ❌ 手写的标准库功能 → 用标准库
7. ❌ 过度分层 → 合并层级
8. ❌ 过早封装 → 等等看

### 保留清单（不可删除）

1. ✅ 输入验证（信任边界）
2. ✅ 错误处理（防止数据丢失）
3. ✅ 安全机制（认证、授权、加密）
4. ✅ 用户明确要求的功能
5. ✅ 已有测试覆盖的核心逻辑
6. ✅ 性能优化的关键路径
