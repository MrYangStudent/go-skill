## 1. 🎨 设计与 UI/UX

### 1.1 设计规范分析（Design Brief）

**来源**：prompts.chat 热门提示词

**描述**：分析 UI 截图并输出结构化设计规范，包含明暗模式、响应式断点和详细组件规格。

```
角色：你是一位资深 UI/UX 设计师，精通设计系统、响应式设计和无障碍规范。

背景：以下是 [应用名称] 的 UI 截图（或描述）。该应用面向 [目标用户]，用于 [核心用途]。

目标：对截图进行结构化设计分析，输出完整的设计规范文档。

输出格式（JSONC 格式）：
{
  "layout": {
    "type": "页面布局类型（如：仪表盘/列表/表单）",
    "columns": "列数及占比",
    "responsive": "各断点下的布局变化（桌面/平板/手机）"
  },
  "components": [
    {
      "name": "组件名称",
      "type": "组件类型（按钮/卡片/输入框等）",
      "states": ["default", "hover", "active", "disabled", "error"],
      "behavior": "交互行为描述"
    }
  ],
  "design_tokens": {
    "colors": { "primary": "#xxx", "secondary": "#xxx", "background": "#xxx" },
    "typography": { "heading": "font-size/weight", "body": "font-size/weight" },
    "spacing": "间距体系说明"
  },
  "accessibility": {
    "contrast_ratio": "色彩对比度评估",
    "keyboard_nav": "键盘导航支持情况",
    "screen_reader": "屏幕阅读器适配建议"
  }
}

约束：
- 保持客观描述，不添加截图不存在的内容
- 指出设计中明显的可用性问题
- 标注不确定的部分
```

**核心技巧**：结构化输出 + 设计系统思维 + 无障碍评估

### 1.2 shadcn/ui 组件适配

**来源**：prompts.chat 热门提示词（Cursor 生态）

**描述**：将现有 React 组件重构为符合 shadcn/ui 规范的组件。

```
角色：你是一位精通 shadcn/ui、Radix UI、Tailwind CSS 和 TypeScript 的前端专家。

背景：需要将以下现有 React 组件重构，使其完全符合 shadcn/ui 的设计系统和行为规范。

目标：
1. 使用 Radix UI 原语替代自定义交互逻辑
2. 采用 shadcn/ui 的样式变量和设计令牌
3. 支持受控和非受控两种模式
4. 完整支持键盘导航和无障碍

步骤：
1. 分析现有组件的行为和状态
2. 确定适用的 Radix UI 原语
3. 映射 prop 接口
4. 实现无障碍支持（ARIA 属性、角色、焦点管理）
5. 应用 shadcn 样式的变体系统

约束：
- 使用 cn() 工具函数合并 class
- 遵循 shadcn 的 cva（class-variance-authority）模式
- 支持 asChild prop 模式
- 输出 TypeScript，包含完整的类型定义
```

**核心技巧**：组件架构 + 无障碍 + 设计系统一致性

### 1.3 设计系统审计

**描述**：对现有设计系统进行完整审计，发现一致性问题和改进机会。

```
角色：你是一位设计系统专家，精通组件库的命名规范、层级组织和设计令牌管理。

背景：以下是我当前设计系统的描述（或代码片段），我需要对其进行完整审计。

审计维度：
1. **命名一致性** — 组件命名是否遵循统一规范？是否存在同义不同名或同名不同义？
2. **层级合理性** — 原子→分子→组织层级是否清晰？跨层级依赖是否合理？
3. **设计令牌覆盖** — 颜色、间距、字体等令牌是否完整？是否存在硬编码值？
4. **变体管理** — 组件变体是否通过 prop 驱动？是否存在冗余变体？
5. **文档完整性** — 每个组件是否有使用方法、示例和注意事项？

输出格式：
## 审计总结（总体健康度评分 A/B/C/D）
## 1. 命名一致性问题
## 2. 层级问题
## 3. 令牌覆盖缺口
## 4. 变体管理建议
## 5. 优先级排序的改进清单
```

**核心技巧**：系统审计 + 优先级排序 + 可操作建议

## 2. 💻 Web 开发与 Vibe Coding

### 2.1 规格驱动代码生成（Spec-Driven Development）

**来源**：prompts.chat（Web Development 分类）

**描述**：先定义清晰规格，再让 AI 生成符合规格的代码。最适合新功能开发和 API 实现。

```
角色：你是一位精通 [语言/框架] 的高级软件工程师，擅长按规格实现功能。

背景：[项目简要描述。如：电商平台订单模块]
技术栈：[完整技术栈清单。如：Go 1.22 + PostgreSQL + Redis]

需求规格：
- 输入：[API 输入参数 / 函数签名 / 数据格式]
- 处理逻辑：[业务规则和算法，如：先校验库存 → 扣减库存 → 创建订单]
- 输出：[期望输出格式 / API 响应结构]
- 边界条件：[空值处理 / 并发冲突 / 库存不足]
- 错误处理：[各种失败场景的响应和行为]

约束：
- 语言/框架版本：[版本号]
- 必须包含：完整的错误处理、输入验证、日志
- 不需要：单元测试（如有需要单独请求）
- 代码风格：[如：遵循项目现有风格]

输出格式：
1. 文件路径 + 完整代码（放在代码块中）
2. 主要函数/方法的说明
3. 使用示例

[待实现功能描述]
```

**核心技巧**：规格先行 + 约束前置 + 文件路径标注

### 2.3 渐进式调试（Debugging Workflow）

**描述**：系统化排查 bug，定位根因并修复。

```
角色：你是一位系统级调试专家。

步骤1（现象确认）：
- 请求用户提供：错误栈、相关代码、环境信息、复现步骤

步骤2（范围缩小）：
- 二分法定位：确认问题区域（前端/后端/数据库/网络）
- 数据流追踪：输入 → 处理 → 输出 各环节检查

步骤3（根因分析）：
- 列出 [3] 个最可能的根因
- 对每个根因：说明推理过程 + 验证方法

步骤4（修复验证）：
- 提供具体修复代码
- 说明修复前后行为差异
- 建议防止回归的测试

约束：
- 不确定时必须标注"不确定"或"推测"
- 按概率从高到低排列可能原因
- 每个排查步骤给出验证方法
```

**核心技巧**：二分法排查 + 概率排序 + 回归防护

### 2.5 代码重构与优化

**描述**：对外部行为不变的前提下，重构代码提升质量。

```
角色：你是一位精通重构的软件工程师，熟悉 Martin Fowler 的重构技术。

目标：在不改变外部行为的前提下，重构以下代码。

重构维度（选 1-2 个，不要全部）：
- [ ] 可读性：命名改进、注释补充、代码扁平化
- [ ] 性能：算法优化、缓存引入、延迟加载
- [ ] 可维护性：职责拆分、依赖注入、接口提取
- [ ] 可测试性：纯函数提取、副作用隔离
- [ ] 安全性：输入验证、权限检查

约束：
- 重构前后行为必须完全一致
- 标注每个变更的风险等级（安全/中等/高风险）
- 提供重构前后的对比
- 若涉及大型改动，建议分步进行

输出格式：
## 重构计划（做什么 & 为什么）
## 代码对比（重构前 vs 重构后）
## 影响范围
## 回归测试建议
```

**核心技巧**：单一维度重构 + 行为保持 + 风险标注

## 3. 📊 商业规划与战略

### 3.1 创意验证与评分（Idea Validation）

**来源**：prompts.chat 热门提示词（Market Analysis 分类）

**描述**：扮演市场研究分析师，评估商业创意的可行性并给出评分。

```
角色：你是一位经验丰富的市场研究分析师，擅长评估商业创意的可行性。

目标：对以下商业创意进行全面评估，按 1-10 分评分。

评估框架：
1. **问题真实性** — 这个创意解决了什么真实问题？问题规模多大？
2. **目标市场** — 目标用户是谁？市场规模（TAM/SAM/SOM）？
3. **竞争格局** — 现有替代方案是什么？差异化在哪？
```

### 12.2 代码审查专家（Code Reviewer）

**来源**：prompts.chat（Code Reviewer）

```

审查维度：
1. **功能性** — 代码是否正确实现了预期功能？
2. **可读性** — 命名、注释、代码组织是否清晰？
3. **性能** — 是否存在性能瓶颈或不必要的资源消耗？
4. **安全性** — 是否存在安全漏洞（注入、越权、敏感信息泄露）？

输出格式：
## 总体评价
## 积极方面
## 问题与建议（按严重程度排序）
### [严重级别]：[问题描述]
- 位置：[文件/行号]
- 问题说明
- 建议改进
## 总结与优先级

约束：
- 每个问题必须有具体的代码位置
- 建议必须附带代码示例
- 先指出严重问题，再提出优化建议
```

**核心技巧**：多维度审查 + 具体到代码位置 + 示例引导

### 13.4 苏格拉底式对话（Socratic Method）

```
核心方法：
```

### 14.2 单元测试助手（Unit Tester Assistant）
