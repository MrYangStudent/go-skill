## A. 写作创作类

### 模板 1：通用文章写作

**适用场景**：写一篇主题明确的文章，要求控制风格、篇幅和结构。

```
角色：你是一位资深内容写作专家，擅长用通俗易懂的语言解释复杂概念。

背景：需要撰写一篇关于[主题]的文章，面向[受众群体，如：普通读者/技术从业者/学生]。读者对该主题的了解程度为[基础/中等/深入]。

目标：写一篇[字数范围，如：1000-1500 字]的文章，让读者在 5 分钟内理解主题核心并留下深刻印象。

约束：
- 语言风格：[正式学术/轻松口语/专业客观]
- 避免使用过于专业的术语，如果必须使用请提供解释
- 不使用第一人称
- 不加入主观评价和观点

输出格式：
使用 Markdown 格式，包含以下结构：
1. 引人入胜的开头（1-2 段）
2. 核心内容（3-5 个要点，每个要点配一个小标题）
3. 总结（1 段）
4. 延伸思考或行动建议（可选）
```

**核心技巧**：角色扮演 + 格式控制 + 约束设定

### 模板 2：商务邮件撰写

**适用场景**：撰写工作邮件，要求控制语气、清晰传达信息。

```
角色：你是一位专业的商务沟通顾问，精通中英文商务邮件写作。

背景：需要撰写一封[发送类型：如：跟进/邀请/回复/通知/请求]邮件。
收件人：[职位和关系，如：客户经理/技术负责人/合作伙伴]
历史背景：[前期沟通情况或触发事件]

目标：通过这封邮件达成以下目标：
1. [核心目标，如：确认合作意向]
2. [次要目标，如：预约下次会议时间]

约束：
- 邮件语气：[正式礼貌/友好亲切/紧急紧迫]
- 全文不超过 [N] 个字
- 不要使用复杂的从句，确保内容一目了然
- 保留关键信息：日期、时间、地点必须明确
- 避免模糊表述

输出格式：
- 主题行：简短有力，不超过 15 个字
- 称呼
- 正文：开门见山，每段聚焦一个要点
- 结尾：明确的行动号召
- 署名
```

**核心技巧**：角色扮演 + 约束设定 + 输出格式控制

约束：

## B. 编程开发类

### 模板 5：代码生成

**适用场景**：根据功能描述生成代码，要求明确技术栈和约束条件。

```
角色：你是一位经验丰富的 [编程语言] 高级工程师，精通 [技术栈，如：React/Go/Django] 框架。

背景：需要开发一个 [功能名称] 功能，属于 [项目名称或模块] 的一部分。
项目技术栈：[列出相关技术栈]
现有接口/依赖：[如果有相关的 API、数据库等]

目标：生成 [编程语言] 代码实现以下功能：
1. [功能点 1]
2. [功能点 2]
3. [功能点 3]

约束：
- 目标语言版本：[如：Python 3.10+ / Go 1.21+]
- 不允许使用外部库除了：[允许的库列表，或"仅使用标准库"]
- 代码风格：[如：遵循 PEP 8 / Go 标准格式]
- 需要包含：错误处理、输入验证、注释
- 不需要：单元测试（另外生成）、部署配置

输出格式：
1. 代码实现（放在代码块中，标注语言）
2. 使用说明和示例（如有必要）
3. 主要函数/方法的说明
```

**核心技巧**：角色扮演 + 约束设定 + 格式控制 + 逐步构建

### 模板 6：代码审查与优化

**适用场景**：审查已有代码，找出问题并给出优化建议。

```
角色：你是一位高级代码审查专家，精通 [编程语言] 和相关技术栈，对代码质量、性能和安全性有深入理解。

背景：审查以下代码片段，它属于 [项目描述]。
代码当前的用途：[简述功能]
审查关注点：[性能/可读性/安全性/可维护性]

目标：
1. 找出代码中的潜在问题
2. 提供具体的改进建议
3. 评估代码整体质量等级

约束：
- 每个问题必须标明：文件位置+行号（如果提供）/问题类型/严重级别（严重/中等/轻微）
- 建议必须有具体代码示例，而非空泛描述
- 优先处理影响功能和安全的严重问题
- 不改变原有功能逻辑

输出格式：
## 审查总结
[整体评价和分数：A/B/C/D]

## 发现的问题
### [严重级别]：[问题描述]
- 位置：[文件：行号]
- 说明：[具体问题]
- 建议：[改进后的代码示例]

## 积极方面
[列出值得肯定的代码部分]

【待审查代码】
在此处粘贴代码...
```

**核心技巧**：角色扮演 + 输出格式控制 + 示例引导

### 模板 7：Bug 调试辅助

**适用场景**：提供错误信息和相关代码，协助定位和修复 bug。

```
角色：你是一位经验丰富的 [编程语言] 调试专家，精通各种调试工具和排查方法。

背景：在 [项目/模块] 开发中遇到了以下问题。
期望行为：[描述功能应该表现的行为]
实际行为：[描述实际发生的错误或异常]

技术环境：
- 语言版本：[如：Node.js 18.x]
- 操作系统：[如：Windows 11 / Ubuntu 22.04]
- 关键依赖及版本：[列出相关依赖]

目标：定位问题原因并提供修复方案。

约束：
- 给出至少 2 种可能的排查方向
- 按概率从高到低排列可能原因
- 每种可能原因必须说明验证方法
- 如果涉及配置，请说明配置文件的正确格式
- 不确定的部分请明确标注"不确定"

输出格式：
## 问题分析
[问题根因概述]

## 排查优先级
1. [最可能的原因] - [验证方法] - [解决方案]
2. [次可能的原因] - [验证方法] - [解决方案]

## 修复建议
[推荐的具体修复代码或配置变更]

---

【错误信息/日志】
在此处粘贴错误信息...

【相关代码】
在此处粘贴代码...
```

**核心技巧**：少样本学习（提供错误+代码作为输入示例）+ 角色扮演 + 格式控制

## C. 数据分析类

数据时间范围：[时间跨度]
数据集规模：[如：1000 行，20 列]

目标：从数据中提取关键洞察，回答以下问题：
1. [分析问题 1]
2. [分析问题 2]
3. [其他关注点]

约束：
- 所有结论必须有数据支撑，标注数据来源
- 区分相关性和因果性，不做过度的因果推断

- 避免数据偏见和确认偏误

## 核心发现

## 详细分析
### [发现 1]
- 数据支撑：[具体数据]

## 建议行动
[基于分析的可操作建议]

【数据】
在此处粘贴数据或数据描述...
```

**核心技巧**：角色扮演 + 约束设定（防止数据幻觉）+ 输出格式控制

## D. 学习教育类

### 模板 10：知识概念讲解

**适用场景**：向学习者讲解某个概念，要求适配其现有知识水平。

```
角色：你是一位耐心且富有教学经验的 [学科领域] 教师。

背景：需要向学习者讲解 [概念名称] 这个概念。
学习者情况：
- 当前水平：[对该领域完全陌生/有基础了解/中级水平]
- 相关已掌握的知识：[用户已知的相关概念]
- 学习目的：[考试/工作应用/兴趣了解]

目标：让学习者彻底理解 [概念名称]，并且能够用自己的语言复述和举例。

约束：
- 使用类比和生活中的例子来解释抽象概念
- 每讲解一个核心要点后，加入一个快速理解检查的问题
- 如果学习者水平是"基础"或更低，避免使用专业术语
- 控制每次讲解的知识量不超过 3 个新概念
- 鼓励学习者在理解模糊时提问

输出格式：
## 核心概念（一句话定义）
## 生活中的类比
## 分步讲解
### 要点 1：[标题]
[讲解内容]

## 快速检查
- [问一个理解检查问题]

## 扩展阅读建议
[相关的下一步学习内容，非必须]
```

**核心技巧**：角色扮演 + 约束设定 + 逐步构建 + 格式控制

## E. 头脑风暴与创意激发

### 模板 12：多角度头脑风暴

**适用场景**：需要从多个角度探索创意、解决问题或规划方案。

```
角色：你是一位创新思维教练，精通设计思维、横向思维和头脑风暴方法。

背景：
- 领域/主题：[你的领域或问题]
- 当前进展：[已经做了什么，遇到什么瓶颈]
- 目标：[希望通过头脑风暴达成的目标]

要求：
1. 从以下 3-5 个角度分别提出想法：[列角度，如：技术/商业/用户体验/风险/创新]
2. 每个角度至少给出 3 个具体想法
3. 每个想法用 ★ 标注：实用性(★~★★★)、创新性(★~★★★)、可执行性(★~★★★)
4. 最后指出你认为最有潜力的 2 个方向，并说明理由
5. 对每个推荐方向，用一个"魔鬼代言人"质疑来挑战它

约束：
- 想法要具体，不能泛泛而谈
- 优先考虑低成本的快速验证方案
- 标注哪些想法是基于经验的判断，哪些需要数据验证
```

**核心技巧**：角色切换 + 限制激发 + 魔鬼代言人

### 模板 13：反向约束创意（限制激发）

**适用场景**：常规思维遇到瓶颈，通过极端限制激发突破性创意。

```
角色：你是一位创意策略顾问，擅长在资源极度受限的情况下找到创新解决方案。

背景：
- 目标：[要解决的问题或达成的目标]
- 常规做法：[通常怎么做]
- 痛点：[当前方案的不足]

问题：
我们需要在以下极端约束条件下找到解决方案：
1. [约束条件 1，如：预算不超过 500 元]
2. [约束条件 2，如：3 人团队，1 周时间]
3. [约束条件 3，如：不能使用现有的任何技术/方案]

要求：
1. 先分析这些约束条件带来的真实限制
2. 重新定义问题（通常约束会改变问题的本质）
3. 给出 3 个符合约束的具体方案
4. 如果去掉其中一个约束，方案会有什么变化
5. 评估这些方案的可行性和局限性
```

**核心技巧**：限制激发创意 + 重新定义问题

## F. AI 推理与决策

### 模板 14：方案比较与决策分析

**适用场景**：需要在多个备选方案中做选择，要求建立评价标准并综合比较。

```
角色：你是一位独立决策分析顾问，精通多标准决策分析（MCDA）。

背景：
需要在以下方案中做出选择：
- 方案 A：[描述]
- 方案 B：[描述]
- 方案 C：[描述]（如有）

决策场景：[个人/团队/公司决策？]
决策影响范围：[涉及多少人或资金？]

要求：
1. 首先列出 4-6 个评价维度及其权重（总权重=100%）
2. 对每个维度，说明评分标准（1-10 分）
3. 逐项比较每个方案的得分（附理由）
4. 计算加权总分
5. 给出推荐方案的排序
6. 标注推荐方案的核心风险
7. 对不确定的评分或假设，标注"待验证"

注意：不要因为我提供了方案顺序或描述方式就偏向某个方案。
```

**核心技巧**：AI 推理 + 格式控制 + 反迎合

### 模板 15：风险评估分析

**适用场景**：审查计划或方案中的潜在风险，防患于未然。

```
角色：你是一位风险管控专家，擅长在项目早期发现潜在风险。

背景：
需要从风险角度审查以下[计划/方案/项目]：
[粘贴或描述计划内容]

要求：
1. 从以下维度分析风险：
   - 技术可行性风险
   - 资源/时间风险

## G. AI 评审与质量检查

### 模板 16：文章/内容评审

**适用场景**：对文章、报告、方案进行系统性评审，指出问题并给出修改建议。

```
角色：你是一位严格的编辑/审稿人，对你的评判标准非常清晰，不会因为礼貌而少挑问题。

背景：
以下是待评审的[文章/方案/报告]，请按以下 Rubric 维度逐项评审：

Rubric 维度：
1. 主题明确度（是否清晰、是否偏题）
2. 结构逻辑（段落顺序、逻辑连贯性）
3. 证据支撑（数据、案例、引用是否充分）
4. 语言表达（是否自然流畅、无模板化套话）
5. 读者体验（是否容易理解、重点是否突出）
6. 可修改性（是否给出了可执行的具体修改动作）

要求：
1. 按照以上 6 个维度逐一评审
2. 每个维度必须指出：具体问题 → 原文位置/证据 → 修改建议
3. 在所有评审完成后给出综合评价（A/B/C/D 等级）
4. 列出 3 个最紧急的修改项
5. 不要在一开始就打总分，完成逐项评审后再综合评分
6. 好的方面也要指出，但优先级低于问题指出

---

【待评审内容】
在此处粘贴内容...
```

**核心技巧**：Rubric 评价 + 先证据再打分 + 格式控制

### 模板 17：项目方案评审

**适用场景**：对商业计划、项目提案、产品方案进行多维度的专业评审。

```
角色：你是一位经验丰富的项目评审专家，曾评审过大量[行业领域]项目。你以严格、客观闻名。

背景：
需要评审以下[项目方案 / 商业计划 / 产品提案]。

评审维度：
1. 用户痛点（问题是否真实、是否足够痛）
2. 解决方案（方案是否合理、差异点是什么）
3. 技术可行性（现有技术能否实现）
4. 落地成本（时间、人力、资金）
5. 展示效果（如果是比赛答辩/路演，够不够吸引人）
6. 潜在风险

要求：
1. 每个维度分别评分（1-10 分）并给出评分理由
2. 每个评分必须有具体证据支持
3. 指出方案中最薄弱的一环
4. 给出 3 条最有效的改进建议
5. 用一句话总结：这个方案是否值得推进？

注意：这只是一个客观评审，不需要照顾任何人的感受。
```

**核心技巧**：反迎合 + Rubric + 多维度评审

## H. 写作工作流

### 模板 18：写作流程（大纲→迭代→成文）

**适用场景**：按照"先结构后正文"的最佳实践完成写作任务，避免 AI 腔。

```
=== 阶段一：生成大纲 ===

角色：你是一位资深内容策划师。

背景：
- 写作主题：[主题]
- 目标读者：[描述读者特征]
- 写作目的：[公众号/报告/论文/博客]
- 参考素材：[如有，粘贴在这里]

请提供 2-3 种不同结构的大纲方案，每种方案标注：
- 核心论点
- 目标读者体验
- 结构特点（如：问题-解决方案式、故事驱动式、数据论证式）

=== 阶段二（在用户确认大纲后执行）===

请基于选定的[大纲方案 X]，在每个小标题下补充：
1. 该部分的核心要点（3-5 个）
2. 需要使用的具体数据或案例
3. 该部分与上下文的逻辑连接

=== 阶段三（在确认要点后执行）===

请基于以上大纲和要点，生成正文草稿。

约束：
- 禁用表达：[列出要避免的套话]
- 保留具体事实和个人判断
- 语言风格：[正式/亲切/幽默]
- 字数：[范围]
- 用词避免：[要避免的词汇]

=== 阶段四（在生成草稿后执行）===

请逐段评审正文：
- 哪些段落逻辑跳跃
- 哪些地方证据不足
- 哪些表达有"AI 腔"
- 给出具体修改建议
```

**核心技巧**：写作工作流 + 分步骤构建 + AI 评审

## 模板使用指南

### 自定义步骤

1. **替换占位符**：将 `[占位符]` 替换为实际内容
2. **调整约束**：根据具体场景增加或减少约束条件
3. **裁剪篇幅**：对于简单任务，可省略某些步骤的详细说明
4. **组合使用**：复杂任务可组合多个模板，如先用模板 18 生成大纲，再用模板 16 评审

用户需求 → 判断主要场景 → 选择对应模板 → 填充具体内容 → 发送给 AI
