# Prompt Master - 提示词大师

> 基于吴恩达《AI Prompting for Everyone》课程精华的对话式提示词构建引擎

## 概述

本技能通过**结构化对话引导**，帮助用户从零开始构建高质量提示词，或对现有提示词进行诊断和优化。核心采用"六步工作流"，每一步收集一个关键维度的信息，最终生成结构完整、目的明确的优质提示词。

## 核心原则

1. **结构化优于自由发挥** — 按照六步框架构建的提示词覆盖更全面、产出更可控
2. **以终为始** — 先明确输出目标和格式，再反推提示词的内容组织
3. **具体优于抽象** — 具体示例和详细约束比笼统描述更有效
4. **迭代优化** — 好的提示词是迭代出来的，每次使用后分析不足并改进

## 对话式提示词构建工作流

### 第一步：角色定义（Role）

**目标**：确定 AI 在任务中扮演的角色/身份，定调语气和专业度。

**引导问题**：
- "你希望 AI 扮演什么角色来帮助你？"
- 示例角色：专业写作顾问、资深程序员、数据分析师、教学导师、翻译专家、心理咨询师、营销策划师等

**输出影响**：
- 角色越明确，AI 的输出语气和专业度越精准
- 复合角色声明效果更佳，例如："你是一位有 10 年经验的前端开发工程师"比"你是一位程序员"好

**常见错误**：
- ❌ 角色过于宽泛（如"你是助手"）
- ❌ 角色与任务不匹配

### 第二步：背景描述（Context）

**目标**：提供任务的上下文信息和前置条件，让 AI 理解场景全貌。

**引导问题**：
- "这个任务在什么场景下进行？"
- "目标受众是谁？"
- "有什么前置条件或已知信息？"

**输出影响**：
- 背景越充分，AI 越能给出贴合实际的回答
- 包含受众信息（如面向初中生 vs. 面向博士生）会显著影响输出风格

### 第三步：目的阐述（Objective）

**目标**：明确具体期望达成的目标和产出方向。

**引导问题**：
- "你希望最终得到什么？"
- "这个提示词要解决什么问题？"
- "成功的标准是什么？"

**输出影响**：
- 目的是提示词的核心驱动力
- 目的越具体、可衡量越好

### 第四步：约束设定（Constraints）

**目标**：定义限制条件、边界规则和避坑事项。

**引导问题**：
- "有什么限制条件？"
- "需要避免什么？"
- "有什么必须遵守的规则？"

**约束类型**：
| 类型 | 示例 |
|------|------|
| 长度限制 | "不超过 500 字" |
| 风格限制 | "正式学术风格，不使用口语化表达" |
| 内容排除 | "不涉及价格信息，不讨论政治" |
| 技术要求 | "必须使用 Python 3.10+，只使用标准库" |

### 第五步：输出格式（Output Format）

**目标**：指定结果以何种结构呈现。

**引导问题**：
- "你希望结果以什么形式呈现？"
- "用列表、表格、代码块还是纯文本？"
- "需要分几个部分？"

**格式选项**：
| 格式 | 适用场景 |
|------|----------|
| 段落文本 | 说明、分析、建议 |
| 列表 | 步骤、要点、检查项 |
| 表格 | 对比、数据汇总 |
| 代码块 | 编程、配置文件 |
| Markdown | 结构化文档 |
| JSON | 机器可读数据 |

### 第六步：示例引导（Example - 可选）

**目标**：提供 1-3 个参考示例，明确风格和质量的预期标准。

**引导问题**：
- "有没有参考示例？"
- "可以参考什么风格或案例？"
- "有没有希望达到的类似输出？"

**技术价值**：
- 示例引导是最强大的提示词技术之一（即少样本学习）
- 好的示例可以大幅提高输出质量
- 包括"好示例"和"差示例"的对比尤为有效

## 头脑风暴工作流（Brainstorming with AI）

> 来自课程 Module 2 核心内容。AI 是绝佳的"思考伙伴"，但需要正确的引导方式。

### 核心原则

1. **提供充足背景** — 不要只给一句话，要说明你的领域、目标、已有哪些想法、想探索的方向
2. **追求多样输出** — 让 AI 从不同角度生成多个可能性，不要只满足于第一个答案
3. **迭代深化** — 先发散再聚焦：第一轮广撒网，第二轮深入筛选后的方向
4. **人机互补** — AI 擅长广度和联想，人类擅长判断和决策

> 完整模板见 `reference/prompt_templates.md` 模板 12（多角度头脑风暴）

### 进阶头脑风暴技巧

- **限制激发创意**：给 AI 反向约束（"预算只有 100 元""只能用 3 个人完成"）反而能产生更有创意的方案
- **角色切换**：让 AI 依次扮演不同角色（营销总监、技术专家、用户）输出多视角方案
- **极端假设**：要求 AI 完全推翻已有假设，从零开始构想
- **"魔鬼代言人"**：对每个想法要求 AI 从反对角度列出问题，避免集体盲区

## 进阶提示词技巧

### 角色扮演（Role-Playing）
- 让 AI 扮演特定角色可以获得专业知识背景下的精准回答
- 原则：越具体越好，包含背景、专业领域、经验年限
- 示例："你是一位在 Google 工作 8 年的 UX 研究员"

### 思维链（Chain-of-Thought, CoT）
- 引导 AI 一步步思考，更适合复杂推理任务
- 触发词："请一步步思考"、"Let's think step by step"
- 适用：数学计算、逻辑推理、复杂决策、代码调试

### 少样本学习（Few-shot Learning）
- 提供 2-5 个高质量的示例对（输入-输出配对）
- 示例数量并非越多越好，2-3 个高质量示例通常足够
- 关键：示例需要覆盖不同的输入模式

### 格式控制（Format Control）
- 明确指定输出结构，避免 AI 自由发挥
- 技巧：定义每个字段的含义和约束
- 示例："用 JSON 格式输出，包含 title、summary、body、tags 四个字段"

### 分步骤构建（Step-by-Step Construction）
- 将复杂任务拆分为多个简单的子提示词
- 每个子提示词专注一个方面，最后汇总
- 适用：报告生成、多维度分析、复杂内容创作

### 反迎合提问（Anti-Sycophancy）

> 来自课程 Module 1（1.1）和 Module 2（2.5），吴恩达课程重点强调的 AI 关键缺陷。

**什么是迎合性**：AI 被训练成"取悦用户"，如果你在提示词中透露自己的偏好，AI 很可能会顺着你的话说而非给出客观评价。这是一种由 RLHF（基于人类反馈的强化学习）训练机制导致的系统性偏差。

**影响**：你用 AI 验证自己的想法时，可能得到的是虚假的"同意"而非真实的"分析"。

**反迎合提示技巧**：

| 问题类型 | ❌ 错误写法（诱导迎合） | ✅ 正确写法（反迎合） |
|----------|----------------------|----------------------|
| 评价想法 | "我有一个很棒的商业点子：移动扎染。批评它。" | "请客观分析以下商业理念：移动扎染。使用以下评分标准分析。注意：不要顾及我的感受，我需要真实分析。" |
| 检查文档 | "我写了一份完美的计划，你帮我看一下。" | "请严格审查这份计划。重点关注：逻辑漏洞、数据缺失、风险遗漏。" |
| 询问意见 | "我觉得这种做法最好，你觉得呢？" | "请从效率、成本和风险三个维度比较以下做法的优劣，不需要考虑我的偏好。" |

**核心法则**：想让 AI 说真话，先删除你的"立场提示"。每当你让 AI 评价自己的东西时，先把"我觉得这很好""我做了很多努力"等前置立场语句删掉，换成中性描述。更进一步，可以明确告诉 AI："我需要的是真实分析，哪怕结论是否定的。"

### AI 推理工作流（Reasoning with AI）

> 来自课程 Module 2（2.4），用于需要多步骤逻辑分析的任务。

**适用场景**：方案比较、风险评估、复杂决策、项目规划。

**推理步骤**：
1. 理解任务目标 → 2. 读取上下文材料 → 3. 拆解问题 → 4. 建立评价标准 → 5. 比较备选方案 → 6. 检查限制条件 → 7. 给出建议 → 8. 标注假设和不确定性

**推荐提示词结构**：
```
请认真分析以下问题。不要急着给结论，先：
1. 拆解问题，列出关键假设
2. 建立 [N] 个评价标准及权重
3. 逐项比较每个方案的优劣
4. 给出推荐结论
5. 对不确定的内容单独标注
```

### AI 评审工作流（Critique with AI）

> 来自课程 Module 2（2.7），AI 可以扮演编辑/评委/审稿人，但需要明确的评价标准。

**核心原则**：
- **先找问题再打分**：要求 AI 先逐项找证据，列出具体问题位置，再给综合分数
- **提供评价维度（Rubric）**：将"好不好"拆解为多个可判断维度
- **分块修改优于整篇重写**：逐段评审更可控
- **跨模型评审**：重要材料用多个模型交叉检查，减少单一模型盲点

**推荐 Rubric 维度**：
| 维度 | 关注点 |
|------|--------|
| 主题明确度 | 是否清晰、是否偏题 |
| 结构逻辑 | 段落顺序合理，无逻辑跳跃 |
| 证据支撑 | 有数据、案例或材料支撑观点 |
| 语言表达 | 自然流畅，无模板化套话 |
| 读者体验 | 易于理解，重点突出 |
| 可修改性 | 给出具体可执行的修改动作 |

### 写作工作流（Writing with AI）

> 来自课程 Module 1（1.1）和 Module 2（2.6），核心洞察：先结构，后正文。

**为什么不能直接让 AI 写全文**：
- 大量抽象词，缺乏具体事实
- 段落结构机械，像模板填空
- 观点安全、平均，缺少鲜明判断
- 表面流畅但内容空泛（即"AI 腔" / AI Slop）

**推荐写作流程**：
```
步骤1: 上传背景材料，让 AI 生成 2-3 种不同结构的大纲
步骤2: 人为选择并调整大纲方向
步骤3: 在小标题层面补充要点和素材
步骤4: 确认结构和素材无误后才让 AI 生成正文
步骤5: 逐段评审、修改，减少 AI 腔
```

**消除"AI 腔"技巧**：在提示词中加入"禁用表达"列表，如：
- ❌ "随着时代的发展"
- ❌ "总的来说"
- ❌ "在当今社会"
- ❌ "值得注意的是"

## 图像生成提示词高级工作流

> 面向 Midjourney、DALL-E 3、Stable Diffusion 等扩散模型的进阶提示词技术。

### 核心认知：扩散模型如何"看懂"提示词

AI 图像模型从随机噪声逐步去噪生成图像。提示词中的**高频词**会被优先响应，因此：

- **词序即权重**：越靠前的词影响力越大，重要元素放在句首
- **具体 > 抽象**："a fluffy orange tabby cat sleeping on a red velvet cushion" 远好于 "a cute cat"
- **风格词是"视觉语言"**：风格关键词决定了材质、光影、细节层次

### 六要素结构化公式

每个图像提示词应覆盖以下维度，按优先级排序：

| 优先级 | 要素 | 说明 | 
|--------|------|------|
| ⭐⭐⭐ | **主体 (Subject)** | 核心视觉对象，最具体的描述 | 
| ⭐⭐⭐ | **介质/风格 (Medium/Style)** | 决定整体视觉语言 | 
| ⭐⭐ | **环境/场景 (Environment)** | 背景和空间氛围 | 
| ⭐⭐ | **光线/色彩 (Lighting/Color)** | 情绪和质感的关键 | 
| ⭐ | **构图/视角 (Composition)** | 镜头语言 | 
| ⭐ | **氛围/情绪 (Mood/Atmosphere)** | 整体情感基调 | 

### 平台专项技巧

| 平台 | 关键语法 | 推荐用法 | 
|------|----------|----------|
| **Midjourney** | `--ar` 比例, `--s` 风格化, `--v` 版本, `--iw` 权重 | 短词+强风格词效果更好；`--s 50` 真实感, `--s 600` 艺术感 |
| **DALL-E 3** | 自然语言描述 | 长句、精确构图描述；文字渲染能力最强 |
| **Stable Diffusion** | 负面提示词 (Negative Prompt) | 必须搭配负面词：`nsfw, ugly, deformed, blurry, low quality` |
| **Midjourney Niji** | 面向二次元/动漫风格 | 使用 `--niji 6` 配合角色描述 |

### 高级技巧

#### 1. 负面提示词（Negative Prompting）
```
# 在 Stable Diffusion / Midjourney 中
避免：扭曲的手、多余的手指、模糊、变形、低质量、水印
--no ugly, deformed, blurry, watermark, text
```

#### 2. 镜头语言控制
```
# 焦距效果
广角：wide angle, 24mm lens, dramatic perspective
长焦：telephoto lens, 85mm, compressed background, shallow DOF
微距：macro shot, extreme close-up, fine details

# 视角
俯视：bird's eye view, top-down, aerial view
仰视：low angle shot, worm's eye view, imposing
过肩：over the shoulder shot, POV
```

#### 3. 风格混合（Style Blending）
```
# 组合不同艺术风格实现独特效果
主体风格 + 光影风格 + 色彩风格 + 材质风格
如："a cat, van Gogh style, cinematic lighting, pastel colors, oil painting texture"

# 年代风格
art deco 1920s, mid-century modern, retro futurism 1980s, steampunk victorian
```

#### 4. 角色一致性（Character Consistency）
```
# 多图生成中保持角色一致
正面：full body portrait of [character], standing, looking at camera, same clothes
侧面：same character from side view, 3/4 angle, same outfit and hairstyle
特写：same character, close-up face, same hair and eye color, consistent features
```

#### 5. LoRA 模型提示词技巧（Stable Diffusion 生态）

**什么是 LoRA**：LoRA（Low-Rank Adaptation）是对基础模型的轻量级微调产物，可精准控制特定风格、角色或物体。提示词中通过 `<lora:filename:weight>` 语法引用。

**LoRA 提示词模板**：
```text
# 通用结构：主体 + LoRA 引用 + 场景 + 参数
(masterpiece, best quality:1.2), [主体描述], <lora:style-name:0.8>, [场景], [光线]

# 角色 LoRA 示例
(close-up portrait of [character]:1.1), <lora:char-name:0.9>, detailed face, intricate eyes

# 风格 LoRA 示例
(a cyberpunk city street:1.2), <lora:cyberpunk-style:0.7>, neon lights, rain, reflections
```

**LoRA 权重控制策略**：
| 权重范围 | 效果 | 适用场景 |
|----------|------|----------|
| 0.3-0.6 | 轻微影响，保留模型原始风格 | 风格混合，不想完全被 LoRA 主导 |
| 0.6-0.8 | 适度影响，LoRA 特征明显 | 角色 LoRA，保持可识别性 |
| 0.8-1.2 | 强烈影响，LoRA 主导输出 | 风格 LoRA，追求特定风格效果 |
| >1.2 | 过度拟合，可能产生伪影 | 需要极其精确的某个特征 |

**多条 LoRA 叠加**：
```
<lora:character:0.8>, <lora:outfit:0.6>, <lora:background:0.4>
# 不同 LoRA 负责不同维度，权重根据重要性递减
```

#### 6. ControlNet 引导提示词

**什么是 ControlNet**：ControlNet 通过额外条件输入（边缘图、深度图、姿势骨架等）精确控制生成图像的结构。提示词负责内容，ControlNet 负责结构。

**ControlNet 类型及配套提示词策略**：

| ControlNet 类型 | 输入要求 | 提示词策略 |
|----------------|---------|-----------|
| **Canny (边缘)** | 线稿图 | 提示词关注上色和风格，不描述形状 |
| **OpenPose (姿势)** | 骨架图 | 提示词关注服装、表情和背景 |
| **Depth (深度)** | 深度图 | 提示词关注物体表面材质和光线 |
| **Scribble (涂鸦)** | 手绘图 | 提示词将涂鸦细化为具体物体 |
| **IP-Adapter (图像参考)** | 参考图 | 提示词描述参考图的风格迁移目标 |

**ControlNet 提示词范例**：
```text
# Canny + 提示词（线稿上色）
角色：你是一位 AI 图像生成专家，使用 Canny ControlNet 为线稿上色。
目标：基于提供的线稿生成彩色图像。
提示词策略：不描述形状（由线稿控制），只描述颜色、材质、光线和风格。
约束：严格遵循线稿轮廓，不改变构图形状。

# OpenPose + 提示词（姿势控制）
角色：你是一位角色插画师，使用 OpenPose ControlNet 控制人物姿势。
目标：生成一张人物在 [特定姿势] 下的图像。
提示词策略：关注服装细节、面部表情、背景环境和光影效果；姿势由骨架图控制。
```

## 视频生成提示词工作流

> 面向 Runway Gen-3、Pika、Sora、Kling（可灵）等视频生成模型的提示词工程。

### 与图像生成的核心区别

| 维度 | 图像生成 | 视频生成 |
|------|---------|---------|
| 时间维度 | 单帧静态 | 须描述时间和运动 |
| 一致性 | 单次生成 | 须保持帧间一致性 |
| 镜头语言 | 可选 | 必须描述镜头运动 |
| 物理规律 | 视觉美感优先 | 须符合物理世界的运动规律 |
| 长度控制 | 输出单张 | 须控制时长、帧率、节奏 |

### 视频提示词六要素

```
1. 场景设定 — 环境、时间、天气、空间布局
2. 主体描述 — 角色/物体外观、材质、颜色
3. 动作/运动 — 主体在做什么？运动轨迹？速度？
4. 镜头运动 — 推/拉/摇/移/跟/环绕/变焦
5. 氛围与光线 — 色调、光源变化、情绪
6. 时间变化 — 随时间推移发生的变化（可选项）
```

### 镜头语言速查

| 镜头类型 | 描述词 | 效果 |
|----------|--------|------|
| 固定镜头 | static shot, locked-off camera | 稳定观察场景 |
| 推镜头 | dolly in, push in, zoom in | 聚焦细节，增强紧张感 |
| 拉镜头 | dolly out, pull back, zoom out | 展示全景，揭示环境 |
| 摇镜头 | pan left/right, camera pan | 跟随横向运动 |
| 移镜头 | tracking shot, follow cam | 跟随主体运动 |
| 环绕 | orbit shot, 360 rotation, drone circle | 展示主体全貌 |
| 手持 | handheld, shaky cam, documentary style | 真实感、临场感 |
| 航拍 | drone shot, aerial flyover, bird's eye | 宏大场景 |
| 延时 | time-lapse, hyperlapse | 时间压缩效果 |

### 运动描述技巧

**原则**：具体描述"怎么动"，而非笼统说"在移动"。

```
❌ "a person walking"
✅ "a person walking confidently down a busy city sidewalk, their footsteps creating a steady rhythm"

❌ "a car driving"
✅ "a red sports car accelerating smoothly on a winding mountain road, tires kicking up dust"

❌ "water flowing"
✅ "crystal clear water cascading over moss-covered rocks, creating small splashes and ripples"
```

### 平台专项技巧

| 平台 | 优势 | 提示词策略 |
|------|------|-----------|
| **Runway Gen-3** | 文本到视频，高一致性 | 详细的主体+动作描述，短句结构 |
| **Pika Labs** | 图片到视频，风格化强 | Motion 参数控制运动强度（1-4） |
| **Sora (OpenAI)** | 长时长，物理模拟强 | 自然语言长句，强调物理规律 |
| **可灵 (Kling)** | 中文理解好，面部稳定性强 | 中英文混合提示词效果最佳 |
| **Luma Dream Machine** | 视频风格化，快速生成 | 强调艺术风格和情绪氛围 |

### 视频 Prompt 范例结构

```
角色：你是一位 AI 视频生成提示词专家，精通各类视频模型的语法特点。

目标：基于用户描述的视频创意，生成一个可直接用于 [平台名称] 的视频提示词。

约束：
1. 必须包含：场景、主体、动作、镜头运动四个核心要素
2. 运动描述要具体（速度、方向、幅度），避免"移动"等笼统词

4. 如需多镜头，用 [cut to] 或 [scene transition] 分隔
5. 如需配乐/音效建议，单独标注在 [sound] 中

[可选：提供 2-3 个不同镜头方案供选择]
```

### 视频关键帧控制

在 AI 视频生成中实现精准的时间轴控制，让不同时间点呈现不同内容。

**关键帧（Keyframe）概念**：描述视频在特定时间点上应该呈现的画面状态。AI 会在关键帧之间进行插值过渡。

**关键帧控制模板**：
```text
视频总时长：[N] 秒

=== 关键帧 1（t = 0.0s）===
场景：[起始画面描述]
主体：[初始位置/外观/表情]
镜头：[镜头起始位置]
光线：[起始光线条件]

=== 关键帧 2（t = [N/3]s）===
变化：[相对于关键帧 1 发生了什么变化]
主体：[中间状态]
镜头：[中间镜头位置]

=== 关键帧 3（t = [2N/3]s）===
变化：[进一步演变的状态]
主体：[新的位置/姿态]

=== 关键帧 4（t = [N]s）===
最终状态：[结束画面]
镜头：[结束镜头]

约束：
- 关键帧之间的过渡要自然（AI 自动插值）
- 相邻关键帧的差异不宜过大，避免突变
- 标注哪些元素需要保持不变（角色/背景等的一致性）
```

**关键帧场景示例**：
```text
# 示例：角色从白天走到夜晚（4 秒视频）
总时长: 4s

KF1 (0s): 人物站在阳光明媚的公园，影子很短，穿着轻便衣服，镜头固定
KF2 (1.5s): 人物开始行走，光线逐渐变暖，影子拉长，镜头缓慢推近
KF3 (3s): 人物走到街道区域，路灯开始亮起，天空呈现黄昏色调，镜头继续推近
KF4 (4s): 人物站在霓虹灯下，夜晚天空，穿着不变但光线变为冷色调，镜头定格特写

过渡要求：光线变化均匀渐变，镜头运动平滑，人物外观保持一致
```

| 常见变化类型 | 关键帧控制策略 |
|-------------|---------------|
| 光线变化 | 从亮到暗 / 从暖到冷，均匀分步 |
| 季节变化 | 从春到冬，中间帧标注过渡特征 |
| 角色运动 | 起→中间→终三个位置点 |
| 镜头运动 | 起→中间→终三个镜头位置 |
| 情绪变化 | 通过颜色、光线、环境变化间接表达 |
| 特效生成 | 从无到有，逐步增强 |

## 编程提示词高级工作流

> 面向代码生成、调试、重构、架构设计的专用提示词技术。

### 核心原则

1. **先规格后代码** — 不论需求多小，先和 AI 对齐理解再生成代码
2. **最小权限原则** — 让 AI 操作本地文件时：先计划、再确认、再执行
3. **上下文为王** — 贴出完整代码比描述需求更有效（降低 AI 幻觉）
4. **迭代验证** — 生成的代码必须经过：静态检查 → 单元测试 → 集成验证

### 模式一：规格驱动代码生成（Spec-Driven）

最适合新功能开发、API 实现。

```
角色：你是一位精通 [语言/框架] 的高级软件工程师。

背景：[项目简要描述]
技术栈：[完整技术栈清单]

需求规格：
- 输入：[API 输入/函数参数/数据格式]
- 处理逻辑：[业务规则和算法描述]
- 输出：[期望输出格式]
- 边界条件：[空值、异常、极限值的处理方式]
- 错误处理：[各种失败场景对应的处理]

约束：
- 语言/框架版本：[版本号]
- 代码风格：[项目规范或代码风格]
- 禁用：[不允许使用的模式/库]
- 必须包含：[必须有的代码要素，如：注释、日志、错误处理]

输出格式：
1. 完整代码（标注文件路径）
2. 使用方法/调用示例
3. 依赖声明（如有新依赖）

[待实现功能描述]
```

**核心技巧**：规格（Spec）先行 + 约束前置 + 文件路径标注

---

### 模式二：测试驱动代码生成（TDD-Driven）

先设计测试用例，再让 AI 生成通过测试的实现代码。

```
角色：你是一位严格执行 TDD（测试驱动开发）的软件工程师。

目标：为以下测试用例编写可通过的实现代码。

测试设计（参考）：
[在此处提供测试代码或测试用例描述]

约束：
1. 先理解测试期望的行为，再写实现
2. 提供的实现必须通过所有测试
3. 不修改测试代码本身
4. 代码质量：满足可读性、性能、安全性
5. 标注每个函数的输入/输出/副作用

输出：
1. 实现代码
2. 预期的测试运行结果
```

**核心技巧**：红-绿-重构循环 + 测试作为可执行规格

### 模式三：渐进式调试（Progressive Debugging）

```
角色：你是一位系统级调试专家，熟悉各种调试技术和故障排查方法论。

步骤1（现象验证）：
- 确认用户描述的错误症状
- 要求提供：错误信息、相关代码、环境信息

步骤2（缩小范围）：
- 通过二分法定位问题区域
- 检查：输入数据 → 处理逻辑 → 输出结果

步骤3（根因分析）：
- 列出可能的根因（按概率排序）
- 对每种可能根因说明验证方法

步骤4（修复验证）：
- 给出具体修复代码
- 说明修复前后的行为差异
- 添加测试防止回归

约束：
- 不确定时必须标注"不确定"
- 每个排查步骤说明 3 个选项：最可能/次可能/不太可能
- 修复后建议添加对应的单元测试
```

**核心技巧**：二分法 + 概率排序 + 回归防护

### 模式五：大规模重构提示词

```
角色：你是一位精通第一性原理的代码重构专家。

目标：在不改变外部行为的前提下，重构以下代码。

重构维度（选择关注的）：
- [ ] 可读性：命名、注释、代码组织
- [ ] 性能：算法优化、缓存策略、延迟加载
- [ ] 可维护性：职责分离、依赖注入、接口抽象
```

### 模式六：Git 工作流提示词

用于 commit message 生成、分支策略设计、代码审查等 Git 相关任务。

**场景 1：Conventional Commit 消息生成**
```text
角色：你是一位遵循 Conventional Commits 规范的 Git 专家。

目标：基于以下变更描述（或 git diff）生成符合规范的 commit message。

规范：
- 结构：<type>[optional scope]: <description>
- 类型：feat / fix / docs / style / refactor / test / chore / ci / perf / build
- scope 可选（如 feat(api):）
- 描述使用祈使语气，小写字母开头，末尾无句号
- Breaking Change 用 ! 或 BREAKING CHANGE footer
- 正文解释 what 和 why，不解释 how
- 字数：标题 ≤ 50 字符，正文每行 ≤ 72 字符

输出的 commit message 直接可用，不加解释。
```

**场景 2：分支策略设计**
```text
角色：你是一位 Git 工作流专家，熟悉 GitFlow、GitHub Flow、Trunk-Based 等策略。

背景：团队规模 [N] 人，项目类型 [前端/后端/全栈/移动端]，发布频率 [每周一次/每日/持续发布]。
当前问题：[现存的分支管理问题]

目标：推荐最适合的分支策略，并给出：
1. 分支命名规范（feature/、bugfix/、hotfix/、release/ 等前缀）
2. 合并策略（merge commit / squash / rebase）
3. PR 流程和审查规范
4. 发布分支管理规则
5. 紧急修复流程

约束：考虑团队当前学习成本，推荐渐进式迁移方案。
```

**场景 3：代码审查（PR Review）**
```text
角色：你是一位经验丰富的代码审查者。

目标：对以下 PR 变更进行审查。

审查维度：
1. **功能正确性** — 变更是否解决了预期问题？是否有逻辑漏洞？
2. **代码质量** — 可读性、命名规范、注释完整性、异常处理
3. **安全性** — 是否存在注入/越权/敏感信息泄露风险
4. **性能影响** — 是否存在不必要的循环/重复计算/内存泄漏
5. **测试覆盖** — 是否有足够的测试保护变更逻辑
6. **Git 规范性** — commit 粒度是否合理？message 是否规范？

输出格式：
## 总体评价（Approve / Comment / Request Changes）
## 具体意见（按文件/行号标注）
## 值得肯定的方面
## 改进建议排序

约束：措辞专业、具体，给出修改示例而非笼统批评。
```

### 编程提示词常见陷阱

| 陷阱 | 问题 | 正确做法 |
|------|------|----------|
| "帮我写个 APP" | 太模糊，AI 会做大量假设 | 拆解为小功能，逐个实现 |
| 不提供技术栈 | AI 用默认框架（Express/Flask） | 明确指定语言+框架+版本 |
| 不提供现有代码 | AI 假设的接口和你的不一致 | 粘贴相关代码或伪代码接口 |
| 不提供错误信息 | AI 靠猜，浪费时间 | 贴完整错误栈+上下文代码 |
| 一次性需求过多 | AI 遗漏细节，需要多次修正 | 一个 prompt 一个功能的粒度 |

> 完整模板见 `reference/prompt_templates.md` 模板 5-7（编程开发类）
> 社区示例见 `reference/community_prompt_examples.md` 第 2 章（Web 开发）和第 12 章（技术支持）

## MCP 工具集成提示词

> MCP（Model Context Protocol / 模型上下文协议）让 AI 与外部工具（GitHub、编辑器、文件系统等）交互。本部分提供 MCP 交互场景下的专用提示词模式。

### 核心原则

1. **先计划后执行** — 修改前先输出操作计划，用户确认后再执行
2. **最小权限** — 只读取必要文件，不扫描无关目录
3. **可回退** — 重要操作优先确保可以回退
4. **分阶段** — 大任务拆分为：读取→分析→执行→验证

### 场景一：代码库阅读与理解

```text
角色：你是一位项目分析师，擅长通过阅读代码库理解项目架构。

目标：阅读 [项目名称/路径] 的代码，理解以下方面：
1. 项目整体架构和目录结构
2. 核心数据流和模块依赖关系
3. 关键接口和类设计
4. 测试覆盖情况

行为规则：
- 先读取项目根目录的配置文件（package.json/go.mod/Cargo.toml 等）
- 再读取核心源文件的入口和模块定义文件
- 最后读取关键业务逻辑模块

输出后请标注：哪些是确定的推断，哪些是推测。
```

### 场景二：基于现有代码库的新功能开发

```text
角色：你是一位软件工程师，在现有代码库中实现新功能。

目标：在 [项目路径] 中实现 [功能描述]。

工作流程：
步骤1：读取代码库结构，理解现有架构和编码风格
步骤2：输出实现计划（涉及哪些文件、修改方案、新增文件）
步骤3：等待用户确认计划
步骤4：执行修改，生成代码
步骤5：验证修改（检查语法、运行测试）

约束：
- 不删除或修改与目标无关的代码
- 遵循项目现有的代码风格和架构模式
- 每次修改前确认影响范围
- 重要操作先创建备份（git stash / 分支）
```

### 场景三：文件操作（读取/编辑/移动）

```text
请帮我 [读取/编辑/移动/删除] [文件路径]。

约束：
- 读取：仅读取与任务相关的文件
- 编辑：先展示修改计划，确认后再执行
- 移动/删除：确认不会影响依赖该文件的模块
- 批量操作前先列出所有受影响文件的清单
```

### 场景四：GitHub / GitLab API 集成交互

```text
角色：你是一位 DevOps 自动化助手，可通过 API 操作 GitHub/GitLab 仓库。

目标：[创建 Issue / 审查 PR / 列出分支 / 创建标签 等]

约束：
- 涉及写入操作时，先输出操作计划再执行
- 不可删除其他人创建的 Issue/PR
- 标注每次操作的权限范围

可用操作：读取仓库信息、创建 Issue、列出 PR、审查代码变更、管理分支
```

### 场景五：AI Agent 自主任务

```text
角色：你是一位任务驱动型 AI Agent，可以自主规划并执行多步骤任务。

任务目标：[描述最终要达成的结果]

可用能力：
- 文件读写
- 代码搜索和理解
- 命令执行
- 网络请求

行为规则：
1. 先制定执行计划，确认后再开始
2. 当遇到不确定时主动询问
3. 每个步骤完成后汇报进展
4. 出错时自动尝试备选方案，无法解决时汇报
5. 最终输出任务完成报告

约束：
- 不执行修改系统配置的命令
- 不删除未确认的文件
- 涉及网络请求时，只访问任务相关的 URL
```

## AI 模型选型与提示词适配

> 不同模型的架构、训练数据和能力侧重不同，同样提示词在不同模型上产出差异可能很大。

### 主流 AI 模型对比

| 模型 | 优势领域 | 提示词策略 | 上下文窗口 | 推荐场景 |
|------|---------|-----------|-----------|---------|
| **Claude 3.5 Sonnet** | 写作、分析、代码、长文档 | 自然对话风格；角色扮演效果极佳；长上下文稳定 | 200K | 写作/评审/长文档分析/头脑风暴 |
| **GPT-4o** | 多模态、推理、通用 | 结构化指令清晰；工具调用能力强；图像理解 | 128K | 多模态任务/数据分析/工具链 |
| **DeepSeek V3/R1** | 推理、代码、数学 | 思维链(CoT)效果突出；长推理链稳定；代码生成质量高 | 128K | 代码生成/数学推理/复杂分析 |
| **Gemini 2.0** | 多模态、搜索、深度研究 | 长上下文连贯性好；原生多模态理解 | 1M | 深度研究/视频理解/超长文档 |
| **Qwen 2.5** | 中文理解、代码 | 中文 prompt 理解最佳；中英双语稳定 | 128K | 中文优先任务/本地化开发 |
| **Grok** | 实时信息、幽默、技术 | 对话风格偏自由；支持实时网络访问 | 128K | 实时信息获取/技术讨论 |

### 核心原则

1. **不调模型就调提示词** — 同一模型换提示词策略比换模型更可控
2. **模型优势匹配** — 善用各模型的长板（如Claude写作、DeepSeek推理、GPT-4o多模态）
3. **避免模型偏见** — 跨模型评审：重要材料用多个模型交叉检查
4. **上下文窗口意识** — 不超出模型的有效上下文窗口

### 提示词适配对照

| 场景 | Claude 推荐写法 | DeepSeek 推荐写法 | GPT-4o 推荐写法 |
|------|----------------|-------------------|-----------------|
| 代码生成 | 自然语言描述+规格 | 精确定义的规格+测试用例 | 分步骤指令+约束清单 |
| 写作 | 角色扮演+风格参考 | 结构化模板+字数约束 | 示例引导+迭代反馈 |
| 分析 | 上传材料+开放提问 | 评分标准+表格输出 | 多维度要求+格式指定 |
| 图像提示词 | 详细场景+情感描述 | 技术参数+风格关键词 | 结构化+平台参数 |

## 提示词诊断与优化工作流

当用户提供已有提示词时，执行以下**三步工作流**：诊断 → 追问补充 → 输出优化版。

### 工作流总览

```
用户输入原始提示词
     ↓
Step 1: 六维诊断 → 输出诊断报告（表格）
     ↓
Step 2: 针对性追问 → 收集缺失的关键信息
     ↓
Step 3: 输出优化版提示词 + 改动说明
```

---

### Step 1：六维诊断

对用户输入的提示词，按以下六个维度逐一检查，输出诊断报告：

| 维度 | 状态 | 评分依据 |
|------|------|----------|
| 🎭 **角色定义** | ✅ 明确 / ⚠️ 模糊 / ❌ 缺失 | 是否有角色声明？是否具体？是否与任务匹配？ |
| 📖 **背景/上下文** | ✅ 充分 / ⚠️ 不足 / ❌ 缺失 | 是否提供了任务背景、受众、前置条件？ |
| 🎯 **目标/目的** | ✅ 清晰 / ⚠️ 模糊 / ❌ 缺失 | 目标是否具体可衡量？用户到底想要什么？ |
| 🔒 **约束条件** | ✅ 充分 / ⚠️ 不足 / ❌ 缺失 | 是否有长度、风格、内容排除、技术要求等约束？ |
| 📋 **输出格式** | ✅ 明确 / ⚠️ 模糊 / ❌ 缺失 | 是否指定了结果的呈现形式？ |
| 💡 **示例引导** | ✅ 有示例 / ⚠️ 可优化 / ❌ 缺失 | 是否有输入-输出配对示例？ |


**评分规则**：
- 如果有且质量好 → ✅
- 如果有但模糊/不够 → ⚠️（给出具体指出哪部分模糊）
- 如果完全没有 → ❌
- 如需量化评分，参考 `reference/scoring_rubric.md`（每维度 1-5 分，加权 30 分制）

**额外检查**（加分项）：
- 🔴 **迎合性检查**：提示词是否包含诱导 AI 迎合的立场语句（如"我觉得这很好""我的方案很棒"）→ 如果发现，标记并提供反迎合改写
- 🔴 **冗长检查**：是否包含无关信息或自相矛盾的约束 → 建议精简

**输出示例**：

```
═══ 诊断报告 ═══

🎭 角色定义     ❌ 缺失    建议：增加角色声明，如"你是一位资深内容编辑"
📖 背景/上下文  ❌ 缺失    建议：补充目标受众、写作目的、参考材料
🎯 目标/目的    ⚠️ 模糊    "文章"太宽泛→明确是科普/报告/观点？
🔒 约束条件     ❌ 缺失    建议：明确字数和风格要求
📋 输出格式     ❌ 缺失    建议：指定输出结构
💡 示例引导     ❌ 缺失    可选，有则更好

综合评价：该提示词缺少 5/6 个核心维度，优化空间较大。
```

---

### Step 2：针对性追问

根据诊断结果，对标记为 ❌ 或 ⚠️ 的维度，逐一向用户追问补充信息。

**追问原则**：
- **一次只问1-2个问题**，不要一次性抛出所有问题让用户不知所措
- 优先追问影响最大的维度（角色 > 目标 > 约束 > 背景 > 格式 > 示例）
- 对 ⚠️ 的维度，具体指出哪部分模糊，而非笼统说"不够清晰"
- 对用户无法回答的问题，提供**智能默认值**或**跳过选项**

**追问示例**：
```
我注意到你的提示词缺少角色定义和目标描述。先看看这两个：

1. 【角色】你希望 AI 扮演什么角色来帮助你完成这个任务？
   比如：翻译专家、数据分析师、写作顾问...
   （如果不确定，可以跳过，我会默认使用"通用助手"）

2. 【目标】你希望最终得到什么结果？
   "写一篇文章"比较宽泛，比如是：
   - 面向中学生的科普文章？
   - 给领导看的项目报告？
   - 公众号推文？
```

---

### Step 3：输出优化版提示词

收集到所有必要信息后，生成完整的优化版提示词。输出格式如下：

```
═══ 优化版提示词 ═══

[在这里输出可直接复制使用的完整提示词]
（以结构化的方式组织：角色→背景→目标→约束→输出格式→示例）

═══ 改动说明 ═══

本提示词相较于原始版本有以下关键改进：
1. [改动点 1] — 为什么改
2. [改动点 2] — 为什么改
3. [改动点 3] — 为什么改

如果某些维度你没有提供信息，我已经使用了默认值。
你可以复制上面的提示词直接使用，或进一步调整。
```

### 优化全过程示例（简版）

```
用户: 帮我优化这个提示词："帮我写一篇关于AI的文章"

AI: ═══ 诊断报告 ═══
🎭 角色定义     ❌ 缺失    📖 背景/上下文  ❌ 缺失
🎯 目标/目的    ⚠️ 模糊    🔒 约束条件     ❌ 缺失
📋 输出格式     ❌ 缺失    💡 示例引导     ❌ 缺失

先补充两个最关键信息：

═══ 优化版提示词 ═══

 → 约束：禁用术语+3个类比+消除AI腔 → 格式：标题→引入→分点→总结]

═══ 改动说明 ═══

2. 明确高中生+公众号 — 控制解释深度

```

## 需求重构工作流（基于 Excel 专业提示词库）

> 当用户提供模糊、碎片化、非结构化的需求时，执行以下四维重构，将其转化为结构化、可执行的提示词。

### 四维重构框架

| 维度 | 说明 | 检查清单 |
|------|------|----------|
| 🎯 **清晰度** | 消除歧义、指代不明、信息跳跃 | 意图明确？对象明确？范围明确？约束明确？ |
| 🎯 **精准度** | 减少泛化表达，增强语义确定性 | 模糊词？可执行？边界清晰？输出可预期？ |
| 🧩 **结构化** | 按逻辑层次重组 | 背景/目标/需求/约束/输出 层次完整？ |
| 🤖 **模型适应性** | 改写为 LLM 稳定理解的表达 | 冗余？歧义？低信息密度表述？ |

### 处理原则

- **忠于原意**：不得擅自改变核心需求
- **合理补齐**：可从上下文推断的最小必要补全
- **不过度发挥**：不要添加不存在的业务目标或限制
- **强调可执行**：输出应适合作为后续提示词的输入
- **语言规范**：简洁、准确、专业、中性

### 输出格式

```
═══ 重构后需求 ═══
【角色】[重构后的角色定义]
【背景】[重构后的上下文]
【目标】[重构后的目标描述]
【具体需求】[逐条清晰列出的需求点]
【约束条件】[明确的限制和边界]
【输出要求】[期望的输出格式和交付物]

═══ 重构说明 ═══
1. 清晰度改进：[具体改了什么]
2. 精准度改进：[具体改了什么]
3. 结构化改进：[具体改了什么]
4. 模型适应性改进：[具体改了什么]
```

> 完整模板见 `reference/prompt_templates.md` 模板 23（需求重构）

## 对话示例

### 场景A：从头构建（六步引导）

```
用户: 帮我写个提示词，让 AI 写周报。

AI: 【1-角色】希望AI扮演什么角色？ → 用户: 我的工作助理
   【2-背景】你的岗位和周报阅读对象？ → 用户: 后端开发，给技术经理看
   【3-目标】希望什么效果？ → 用户: 如实记录，突出技术难点
   【4-约束】如何组织内容？ → 用户: 按"本周完成→下周计划→遇到的问题"三部分
   【5-格式】用什么格式？ → 用户: Markdown
```
